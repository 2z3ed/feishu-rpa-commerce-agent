# P73 治理主线收口评审

> 结论：**通过收口评审，P73 可以正式收口。**  
> 适用范围：仅 `warehouse.adjust_inventory` 单动作治理主线。  
> 说明：本文件是 P73 的阶段收口与交接索引，不是功能开发文档。

---

## 1. 阶段定义

P73 的阶段目标，不是新增第二动作，也不是进入正式生产接入，而是把 P72 之后的单动作主线推进为一条**可门禁、可留痕、可回放、可回归、可升级判断**的治理主线。

P73 的目标可概括为：

- 定义清楚主线边界
- 固化最小治理清单
- 固化最小验收模板
- 固化最小开发清单与实施顺序
- 落地门禁检查入口
- 落地门禁最小留痕
- 对齐固定样本回归矩阵
- 固化升级准入检查项

---

## 2. 各轮分别完成了什么

### P73 正式主线定义

- 明确 P73 不是生产接入轮，不是第二动作轮，也不是页面逼真化轮
- 明确 P73 是单动作主线的门禁 / 留痕 / 演练深化阶段
- 明确冻结边界与升级准入前提

### P73.1 最小治理清单收敛

- 收敛出四类最小治理清单：
  - 门禁点清单
  - 留痕字段清单
  - 演练 / 回归矩阵清单
  - 升级准入条件清单

### P73.2 最小验收模板 / 检查表收敛

- 将四类治理清单压成最小验收模板
- 补齐每类模板的检查方法
- 明确当前事实与建议口径的区分

### P73.3 最小开发清单 / 实施顺序定义

- 明确未来开发的最小任务清单
- 明确先后顺序
- 明确每项任务的最小改动面与最小通过标准

### P73 开发第一轮：固化门禁检查入口

- 在 `app/graph/nodes/execute_action.py` 中为 `warehouse.adjust_inventory` 落地最小门禁入口
- 实现 `allow / blocked / reason` 三态表达
- 保持 confirm 唯一放行语义不变

### P73 开发第二轮：留痕字段最小落点

- 将门禁结果最小落点进入现有结果链 / 证据链
- 以最小字段方式进入 `action_executed` 详情：
  - `gate_allow`
  - `gate_status`
  - `gate_reason`
- 不重做 `/tasks` / `/steps`

### P73 开发第三轮：固定样本回归矩阵对齐

- 对齐 4 个固定样本：
  - `success`
  - `element_missing`
  - `page_timeout`
  - `verify_fail`
- 统一样本检查结构与回归断言
- 修正 `gate_*` 语义回到前置门禁结果

### P73 开发第四轮：升级准入检查项固化

- 固化 7 条升级准入检查项
- 为每条准入项补最小检查方法
- 保持边界冻结，不扩展生产接入

---

## 3. 已完成能力

P73 当前已完成的能力可以定性为：

### 3.1 单动作治理边界已明确

- 仍然只围绕 `warehouse.adjust_inventory`
- 不扩第二个动作
- 不接正式生产
- 不接影刀控制台 / API Key / Flow
- 不让飞书直接触发影刀

### 3.2 门禁能力已落地

- 门禁入口已进入代码
- 门禁结果可表达 `allow / blocked / reason`
- 门禁结果已最小化进入现有链路

### 3.3 留痕能力已落地

- 门禁结果可在现有 `action_executed` 详情中稳定出现
- 不需要重做 `/tasks` / `/steps`

### 3.4 回归矩阵已落地

- 4 个固定样本已统一结构
- 样本断言已对齐
- `gate_*` 语义已修正并稳定

### 3.5 升级准入判断已落地

- 7 条升级准入检查项已固化
- 每条都有最小检查方法
- 可用于判断是否进入下一阶段

---

## 4. 当前明确没做什么

P73 当前明确没有做：

- 没有扩第二个动作并实现
- 没有接真实 Odoo 生产页面
- 没有接影刀控制台 / API Key / Flow
- 没有让飞书直接触发影刀
- 没有做正式生产接入
- 没有做大重构
- 没有重做 `/tasks` / `/steps`
- 没有新增一套完整治理字段体系
- 没有把单动作主线升级为生产线

---

## 5. 为什么现在可以正式收口

P73 现在可以正式收口，原因是它已经完成了该阶段的核心闭环：

1. **定义完成**：P73 阶段目标和边界已经明确
2. **清单完成**：最小治理清单已收敛
3. **模板完成**：最小验收模板已固化
4. **开发顺序完成**：最小开发清单与实施顺序已定义
5. **入口完成**：门禁检查入口已落地
6. **留痕完成**：门禁结果已最小落点
7. **回归完成**：固定样本矩阵已对齐
8. **准入完成**：升级准入检查项已固化

这意味着 P73 已经不再只是定义层，而是达成了一个稳定的、可交接的单动作治理阶段。

---

## 6. 为什么当前仍然不是正式生产接入

尽管 P73 已可收口，但它仍然不是正式生产接入，原因很明确：

- 当前仍只围绕单动作治理主线
- 当前仍未接正式生产页面
- 当前仍未接影刀控制台 / API / Flow
- 当前仍未建立生产级权限 / 运维 / 安全接入链路
- 当前的最小门禁、留痕、回归、准入仍属于治理与受控验证语境

因此，P73 的收口不是生产上线，而是**治理主线阶段性完成**。

## 9. 通过结论

**P73 通过收口评审，可以正式收口。**

收口的含义是：

- P73 不再继续拆开发轮次
- P73 不再继续补纯文档层级
- P73 作为单动作治理主线阶段已完成
- 后续如有新主线，应以新的阶段定义重新起立

---

## 10. 后续主线入口

P73 收口之后，如果需要继续推进，入口不应是继续 P73 文档，而应是：

- 重新评估下一条主线的定位
- 先确认是否仍处于非生产、单动作治理语境
- 再决定是否进入更高一级阶段定义

在没有新的阶段定义前，P73 应视为已收口完成。
