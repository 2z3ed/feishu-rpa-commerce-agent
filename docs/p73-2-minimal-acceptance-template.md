# P73.2 最小验收模板 / 检查表

> 状态：定义层模板文档，非功能开发文档。  
> 适用范围：仅 `warehouse.adjust_inventory` 单动作主线。  
> 当前定位：把 P73.1 的四类最小治理清单压成可执行、可检查、可复用的最小验收模板。  
> 明确边界：不做功能开发，不接正式生产，不扩第二动作，不接影刀控制台 / API Key / Flow，不让飞书直接触发影刀，不做大重构。

---

## 1. 本轮目标

P73 已正式定义主线，P73.1 已收敛四类最小治理清单。P73.2 的任务不是新增能力，而是把这些清单压成统一的最小验收模板，让后续检查可以用同一套口径判断：看什么、怎么查、什么算通过、什么算未通过。

本轮只处理四类模板：

1. 门禁点最小验收模板
2. 留痕字段最小验收模板
3. 演练 / 回归矩阵最小验收模板
4. 升级准入条件最小验收模板

---

## 2. 最小门禁验收模板

### 模板内容

**检查对象**：门禁点是否已被明确且可执行。

**最小检查项**：

- 是否明确任务入口门禁
- 是否明确编排门禁
- 是否明确执行前门禁
- 是否明确页面门禁
- 是否明确结果门禁
- 是否明确回放门禁
- 是否明确治理门禁

**通过标准**：

- 每个门禁点都有明确名称
- 每个门禁点都有明确触发位置或检查位置
- 每个门禁点都有明确“通过 / 拦截”的判断含义
- 门禁点之间没有重复定义或职责冲突

**未通过标准**：

- 门禁点名称模糊
- 触发位置不明确
- 判断结果不明确
- 同一门禁职责与其它门禁重叠

### 检查方法

- **看什么**：看治理定义文档、清单文档中是否列出门禁点
- **怎么查**：逐项核对是否存在“名称 + 位置 + 判定”三要素
- **看到什么算通过**：七类门禁均已被点名，且没有职责冲突
- **看到什么算未通过**：缺项、重复项、或只有建议没有判断口径

## 3. 最小留痕字段验收模板

### 模板内容

**检查对象**：留痕字段口径是否已明确到最小可复用集合。

**最小检查项**：

- `/tasks` 是否定义任务级结果与治理字段
- `/steps` 是否定义步骤级证据与失败码
- 证据目录是否定义原始结果与回放关联
- 演练记录是否定义样本、日期、结论、门禁命中情况

**通过标准**：

- 每类留痕载体都已明确其职责
- 已区分任务级、步骤级、证据级、演练级口径
- 尚未实现的字段只作为建议固化项或待落地项出现

**未通过标准**：

- 把建议字段写成已实现事实
- 留痕职责混淆
- 没有区分任务级与步骤级口径
- 没有说明哪些是建议项、哪些是当前事实

### 检查方法

- **看什么**：看是否存在留痕载体清单及字段归属说明
- **怎么查**：按 `/tasks`、`/steps`、证据目录、演练记录四类逐一核对
- **看到什么算通过**：四类载体都有清晰职责，且事实/建议分层清楚
- **看到什么算未通过**：字段归属不清、口径混写、或把建议当事实

---

## 4. 最小演练 / 回归矩阵验收模板

### 模板内容

**检查对象**：固定样本是否形成可回归、可复查的最小矩阵。

**最小检查项**：

- 是否覆盖 `success`
- 是否覆盖 `element_missing`
- 是否覆盖 `page_timeout`
- 是否覆盖 `verify_fail`
- 是否说明每个样本的触发方式
- 是否说明每个样本的主断言
- 是否说明每个样本的期望失败层
- 是否说明是否允许回放
- 是否说明是否进入 `/steps` 留痕

**通过标准**：

- 四个固定样本均已列出
- 每个样本都有最小判定要素
- 每个样本的输出结果能够与门禁、留痕口径对齐

**未通过标准**：

- 样本不完整
- 没有触发方式
- 没有主断言
- 没有失败层对应关系
- 不能与 `/steps` 或门禁口径对齐

### 检查方法

- **看什么**：看回归矩阵清单和样本说明
- **怎么查**：逐个样本检查触发方式、断言、失败层、回放权限、留痕要求
- **看到什么算通过**：四类样本完整且结构一致
- **看到什么算未通过**：样本缺失、维度缺失、或口径不统一

---

## 5. 最小升级准入条件验收模板

### 模板内容

**检查对象**：是否已经达到可以讨论更高一级动作的最低门槛。

**最小检查项**：

- 单动作主线门禁点是否稳定
- `/tasks`、`/steps`、证据目录、演练记录口径是否稳定
- 固定样本回归矩阵是否稳定
- `--task-id` 回放与样本对照是否稳定
- 失败层 / 主原因映射是否稳定
- 是否仍然保持冻结边界
- 是否无需推翻已有主线语义

**通过标准**：

- 上述 7 项都已明确为稳定项
- 每一项都能被检查或复核
- 没有把未完成项误写为已完成

**未通过标准**：

- 任一核心口径未稳定
- 仍有未定义或冲突项
- 还需要重写主线语义才能继续讨论下一阶段

### 检查方法

- **看什么**：看升级准入条件清单
- **怎么查**：逐条核对是否具备稳定性结论
- **看到什么算通过**：7 条准入条件都可被明确判断为已稳定
- **看到什么算未通过**：仍停留在“待确认”“建议”“可能”层面

## 6. 最小使用顺序

如果后续有人拿这套模板去实际检查，建议顺序如下：

1. **先看门禁验收模板**：先确认是否允许进入主线检查
2. **再看留痕字段验收模板**：确认是否可追踪、可留痕
3. **再看演练 / 回归矩阵验收模板**：确认是否可回放、可复核
4. **最后看升级准入条件验收模板**：确认是否可以讨论更高一级主线

这个顺序的原则是：先放行条件，再证据条件，再样本条件，最后升级条件。
