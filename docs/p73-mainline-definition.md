# P73 单动作主线治理加固定义

> 状态：P73 主线定义阶段。  
> 固定动作：仅 `warehouse.adjust_inventory`。  
> 当前定位：单动作主线的门禁 / 留痕 / 演练深化。  
> 明确边界：必须只围绕 `warehouse.adjust_inventory`，必须不接正式生产，必须不接影刀控制台 / API Key / Flow，必须不让飞书直接触发影刀，必须不扩第二个动作。

---

## 1. P73 为什么存在

P6.2 / P70 / P71 / P72 已正式收口后，下一步最重要的事情不是继续扩功能，而是把**单动作主线**的治理能力固定下来。

P73 的目的不是新增执行能力，而是把现有主线固定成一条更稳定、可交接、可回归、可升级的治理主线。

P73 继续只围绕 `warehouse.adjust_inventory`，因为当前主链已经冻结为：

飞书消息 → 幂等 → 建任务 → Celery 异步执行 → LangGraph 编排 → API / confirm / RPA / bridge 执行链 → 回飞书结果 → `/tasks` 与 `/steps` 可查询、可追踪、可留痕。

P73 只做这条链路上的治理固化，不开新动作，不换主线，不跳生产。

---

## 2. 为什么选这条主线

当前唯一推荐主线必须是：**单动作主线的门禁 / 留痕 / 演练深化**。

选择它的原因必须固定为：

- 最贴近已收口的单动作主线，不需要重建语义
- 最不容易破坏 P6.2 / P70 / P71 / P72 的既有边界
- 最适合在正式生产接入前继续夯实基础
- 最能把 `/tasks`、`/steps`、证据目录和演练记录变成长期可继承资产
- 最不容易因为范围扩张导致返工

P73 的核心不是“再做一轮验证页面”，而是“把现有主线治理成必须稳定推进的基础设施”。

---

## 3. 为什么不选其它三条候选主线

### 3.1 不选“更接近真实后台的进一步受控验证”

**原因：**

- P72 已经完成了更真实受控页面的正式收口，继续往这个方向走，容易重复页面逼真化工作
- 风险是把主线拉回到“再做一轮受控验证”循环里，增量不如治理深化稳定
- 容易在页面差异、样本差异、证据差异上产生返工

### 3.2 不选“第二个动作复制验证”

**原因：**

- 这会直接扩第二个动作，触碰当前冻结边界
- 会引入新的失败语义、门禁差异、回放差异和留痕差异
- 可能迫使回头调整当前单动作抽象，不符合本轮不重构要求

### 3.3 不选“正式生产接入前的权限 / 运维 / 安全准备”

**原因：**

- 这条线重要，但时机偏早
- 当前仍明确处于非生产接入边界
- 如果过早推进，容易把治理升级与生产化准备混在一起，导致返工和范围膨胀

因此，当前最合理的唯一主线仍是：**先把单动作主线的门禁、留痕、演练和回归固化好。**

---

## 5. 本轮四项交付物

### 5.1 当前已确认规则：单动作主线的门禁点清单

当前已确认规则之一，必须梳理并固定以下门禁类型：

- **任务入口门禁**：消息是否幂等、任务是否可建
- **编排门禁**：LangGraph 是否能进入正确执行路径
- **执行前门禁**：confirm / API / RPA / bridge 的前置条件是否满足
- **页面门禁**：页面是否可定位、可操作、可验证
- **结果门禁**：结果是否能回写、能留痕、能追踪
- **回放门禁**：`--task-id` 是否能稳定重放并对照 `/steps`
- **治理门禁**：失败层、失败码、主原因是否稳定映射

### 5.2 当前已确认规则：`/tasks`、`/steps`、证据目录、演练记录的留痕字段口径

当前已确认规则之一，必须固化哪些字段是主线必须留痕、哪些字段是辅助字段。固定口径如下：

- **任务级留痕**：
  - 任务来源
  - 幂等标识
  - 动作名
  - 执行状态
  - 主失败层
  - 主失败原因
  - 最终结果

- **步骤级留痕**：
  - 执行步骤名
  - 页面 URL
  - 页面 profile
  - 页面 steps
  - 页面证据数量
  - 页面失败码
  - 断言结果

- **证据目录留痕**：
  - 原始结果路径
  - 关键证据文件路径
  - 回放关联 ID
  - 样本类型

- **演练记录留痕**：
  - 演练日期
  - 演练样本
  - 演练结论
  - 是否命中门禁
  - 是否与回归矩阵一致

### 5.3 当前已确认规则：单动作主线的演练 / 回归矩阵主视图

当前已确认规则之一，必须形成一张主视图，覆盖以下内容：

- success
- element_missing
- page_timeout
- verify_fail
- 旧主线不回归

每个样本至少明确：

- 触发方式
- 主断言
- 失败层期望
- 是否允许回放
- 是否进入 `/steps` 证据

### 5.4 下一轮待落地治理清单：进入“更接近生产”或“扩第二动作”前的升级准入规则

下一轮待落地治理清单之一，必须定义升级前的最低门槛，例如：

- 单动作主线的门禁点稳定
- `/tasks` 与 `/steps` 留痕口径稳定
- 固定样本回归稳定
- 回放与演练记录一致
- 失败层 / 失败码映射稳定
- 不破坏当前冻结边界

只有满足这些条件，才允许讨论更接近生产的准备或第二动作复制验证。
