# P6.0 阶段收口文档：Odoo readonly 主线复制验证（`warehouse.query_inventory`）

> **阶段定位**：P6 = Odoo 主线复制验证；P6.0 =仅验证 **Odoo readonly 一条最小动作** 能否按 Woo 已验证的方法论成立第二遍。  
> **固定动作**：`warehouse.query_inventory`（只读）。  
> **边界**：不扩平台、不扩第二动作、不开 Odoo 写链、不破坏 Woo 已收口主线、不做真实生产写与真实登录自动化。

---

## 1. P6.0 阶段定义

P6.0 是在 **不引入 Odoo 写链与治理全栈** 的前提下，完成：

- 一条 **Odoo readonly** 业务动作从「可识别 → 可执行 → 可观测 → 可复验 → 可防回退」的最小闭环；
- 证明 **Woo 侧已沉淀的方法论**（任务台账、`/tasks`/`/steps`、关键口径字段、样本脚本、SOP、回归）**可迁移到第二平台（Odoo）**。

---

## 2. P6.0 为什么成立（方法论层面）

- **复制方法，不复制冗余复杂度**：先只读、先一条动作、先内部 sandbox 可观测链，避免过早绑定真实 Odoo DOM/会话/写权限。
- **观测面先行**：`/api/v1/tasks`、`/api/v1/tasks/{task_id}`、`/api/v1/tasks/{task_id}/steps` 与 `action_executed.detail` 形成可取证闭环。
- **口径可断言**：`result_summary` 与 `action_executed.detail` 的关键字段集合稳定，才能进入下一阶段（写链/治理）而不「语义漂移」。

---

## 3. 三轮分别做了什么（摘要）

| 轮次 | 主题 | 核心产出 |
|------|------|----------|
| **第一轮** | Odoo readonly 样板链成立 | `warehouse.query_inventory` 可走通任务创建与执行；`/tasks`、`/steps` 可取证；`result_summary` 与 `action_executed.detail` 关键字段可读；最小 SOP + 重复回归路径 |
| **第二轮** | 口径收紧与防回退 | `execution_mode` 对该 readonly 链语义收紧为 **`api`**（避免被旧 `mock` 误导）；summary 模板锚点更稳定；最小测试资产卡住语义回退 |
| **第三轮** | detail 固化与收口版 SOP | `action_executed.detail` 采用 **固定必填字段集合 + 固定顺序 + 值归一化**；测试卡住字段缺失/空值/`none`；SOP 增补硬检查项与常见失败示例 |

---

## 4. 当前已经成立的结果（交接版「过线清单」）

下一位接手人 **不看聊天** 也可据此判定「P6.0 已过」：

1. **动作范围**：仅 `warehouse.query_inventory`（Odoo readonly），无第二动作。
2. **执行语义**：该链路的 `execution_mode` 口径为 **`api`**（与 internal sandbox + adapter + mapper 的样板链一致）。
3. **用户可见摘要**：`result_summary` 稳定、含固定模板锚点（含 intent 前缀 `[warehouse.query_inventory]`等）。
4. **步骤取证**：`/steps` 中 `action_executed.detail` **固定包含**且顺序稳定、值可断言的字段集合（至少）：
   - `execution_mode`
   - `provider_id`
   - `capability`
   - `readiness_status`
   - `endpoint_profile`
   - `session_injection_mode`
5. **复验资产**：
   - 手动样本：`scripts/p60_odoo_readonly_sample.py`
   - 收口版 SOP：`docs/p60-odoo-readonly-sop.md`
   - 防回退测试：含 Odoo detail / execution_mode / summary 相关用例（见 `tests/` 中 P6.0 相关文件）

---

## 5. 当前明确没做什么（边界声明）

- 未开 **Odoo 第二动作**；未扩 **Chatwoot**；未加第三个业务动作。
- 未进入 **Odoo 写链**；未做 **真实生产写**；未做 **真实登录自动化**。
- 未重做 **飞书入口主链**；未重写 **confirm 体系**；未做大框架级 **execute mode** 重构。
- **未破坏** Woo 已收口主线（P6.0 变更范围刻意收敛在 Odoo readonly 与观测/文档/测试）。

---

## 6. 为什么现在先收口而不是继续留在 P6.0 开发

- P6.0 的验收目标是 **「方法论可迁移」**，不是 **「Odoo 功能堆叠」**；三轮已把样板链、口径、detail、SOP、回归资产钉死。
- 继续在 P6.0 内开发会自然滑向 **扩动作 / 提前写链 / 提前治理**，违背「先只读闭环、再高风险写」的总原则。
- 收口后进入 P6.1，才能把 **变更类型** 从「只读复制验证」切换为「写链与风险」，避免阶段混淆。

---

## 7. 当前阶段通过结论

**P6.0 正式判通过。**

依据：Odoo readonly 固定动作 `warehouse.query_inventory` 已形成 **可执行 + 可观测 + 可复验 + 可防回退** 的阶段性成果，且阶段边界清晰、未越界扩平台/扩动作/开写链。
