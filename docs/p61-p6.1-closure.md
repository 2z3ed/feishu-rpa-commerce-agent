# P6.1 阶段收口文档：Odoo 最小高风险写链闭环（`warehouse.adjust_inventory`）

> **阶段定位**：P6 = Odoo 主线复制验证；P6.1 = 在 P6.0 readonly 基线之上，复制 Woo 方法论到 Odoo，形成 **1 条最小高风险写链闭环**。  
> **固定动作**：`warehouse.adjust_inventory`（高风险写，受控 sandbox）。  
> **边界**：不扩第二个 Odoo 写动作；不扩 Chatwoot；不加第三个业务动作；不破坏 Woo 已收口主线；不做真实生产写与真实登录自动化；不重写 confirm 体系与 execute mode 大框架。

---

## 1. P6.1 阶段定义

P6.1 只做一件事：把 Odoo 的一个最小高风险写动作做成完整闭环：

**发起任务 → awaiting_confirmation → `system.confirm_task` 唯一放行 → 受控写链 → 写后核验 → `/tasks` 与 `/steps` 可完整取证**

> 说明：P6.1 不是开放真实 Odoo 写权限；不是治理/门禁；不是扩动作；只验证“高风险写链 + confirm + 写后核验”的可迁移性与可取证性。

---

## 2. 为什么是 `warehouse.adjust_inventory`

选择库存调整作为样板动作的理由（与阶段目标对齐）：

- **高风险且语义清晰**：库存写入属于高风险操作，天然需要二次确认与写后核验。
- **最小可控**：可以用 internal sandbox 模拟写入，不引入真实 Odoo 登录/权限问题。
- **可取证**：写前值、目标值、写后值与核验结果都可以固化到 `/steps` 与 `action_executed.detail`，便于后续进入治理阶段。

---

## 3. 三轮分别做了什么（摘要）

| 轮次 | 主题 | 核心产出 |
|------|------|----------|
| **第一轮** | 最小闭环成立 | 新增 `warehouse.adjust_inventory`；原始任务进入 `awaiting_confirmation`；`system.confirm_task` 成为唯一放行入口；confirm 后走 internal sandbox 受控写链；写后核验（写前/写后/verify_*）；`/tasks` 与 `/steps` 可取证；最小样本脚本 + SOP + 回归 |
| **第二轮** | confirm 语义收紧 + 缺 risk_context 负例固化 | `action_executed.detail` 固定补 `confirm_backend=internal_sandbox`（可解释 confirm 放行后的受控写链）；缺 `risk_context` 时 confirm 必须失败且口径稳定；证明 confirm 不回退到 summary 反解析；补负例测试与 SOP |
| **第三轮** | 结构化上下文校验收紧 | `risk_context` 必须 **合法 JSON + dict 形状 + 关键字段齐全** 才能放行；risk_context 缺失 / 非法 JSON / 缺关键字段均明确失败并可取证；补对应测试与 SOP 负例说明 |

---

## 4. 当前已经成立的结果（交接版「过线清单」）

下一位接手人 **不看聊天** 也可据此判定「P6.1 已过」：

1. **动作范围**：仅 `warehouse.adjust_inventory`（Odoo 高风险写），无第二个 Odoo 写动作。
2. **状态机闭环**：原始任务先 `awaiting_confirmation`，不会直接 `succeeded`。
3. **唯一放行入口**：`system.confirm_task` 是唯一放行入口；confirm 只消费 `awaiting_confirmation` 任务。
4. **结构化上下文唯一前提**：confirm 不解析 summary 文案；只读取 `/steps` 的 `risk_context`（JSON）作为放行依据。
5. **受控写链成立**：confirm 后走 internal sandbox 写链（非生产写）。
6. **写后核验成立**：至少可取证字段包含：
   - 写前：`old_inventory`
   - 目标：`delta` / `target_inventory`
   - 写后：`post_inventory`
   - 核验：`verify_passed` / `verify_reason`
   - 结果：`operation_result`
7. **步骤取证稳定**：确认任务 `/steps` 的 `action_executed.detail` 至少稳定包含：
   - `provider_id=odoo`
   - `capability=warehouse.adjust_inventory`
   - `confirm_backend=internal_sandbox`
   - `target_task_id / original_update_task_id / confirm_task_id`
   - `operation_result / verify_passed / verify_reason`
   - 失败路径包含 `failure_layer`
8. **负例闭环成立**：以下情形 confirm 必须阻断并留证据，且不回退到 summary 反解析：
   - risk_context 缺失：`failure_layer=confirm_context_missing`
   - risk_context JSON 非法：`failure_layer=confirm_context_invalid_json`
   - risk_context 缺关键字段：`failure_layer=confirm_context_incomplete`（`verify_reason` 列出缺字段）
9. **复验资产齐全**：
   - 手动样本脚本：`scripts/p61_odoo_adjust_inventory_sample.py`
   - 收口版 SOP：`docs/p61-odoo-adjust-inventory-sop.md`
   - 最小测试资产：`tests/test_p61_odoo_adjust_inventory_flow.py`

- 未破坏 Woo 已收口主线（变更范围限定在 Odoo 高风险写样板链与取证/文档/测试）。

---

## 6. 为什么现在先收口而不是继续留在 P6.1 补开发

- P6.1 的验收目标是 **“最小高风险写链闭环可迁移”**，不是 **“Odoo 写功能堆叠”**。
- 三轮已经把：
  - `awaiting_confirmation` → `system.confirm_task` 唯一放行
  - 结构化上下文（risk_context）作为唯一放行前提
  - 受控写链 + 写后核验
  - `/tasks` + `/steps` 取证口径
  全部钉死并可回归断言。
- 继续在 P6.1 增补小功能，收益递减且容易滑向“扩动作/提前治理”，不符合阶段边界。

---

## 7. 当前阶段通过结论

**P6.1 正式判通过。**

依据：`warehouse.adjust_inventory` 已形成 **可发起 + 必须确认 + confirm 唯一放行 + 受控写链 + 写后核验 + 可取证 + 结构化上下文校验负例** 的阶段性成果，且边界清晰、未越界扩动作/扩平台/开生产写。
