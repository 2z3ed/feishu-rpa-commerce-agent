# P73.1 最小治理清单收敛

> 状态：P73 正式定义之后的最小可执行治理清单。  
> 适用范围：仅 `warehouse.adjust_inventory` 单动作主线。  
> 明确边界：不改代码主链，不扩第二个动作，不接正式生产，不接影刀控制台 / API Key / Flow，不让飞书直接触发影刀，不做大重构，不进入功能实现。

---

## A. 本轮做了什么

本轮把 P73 正式定义文档中的治理项，收敛成四类最小清单：

1. 门禁点清单
2. 留痕字段清单
3. 演练 / 回归矩阵清单
4. 升级准入条件清单

同时为每一类补了最小验收标准，并给出下一轮最小执行顺序。

---

## B. 为什么这样做

P73 已经不再是“要不要做治理”的问题，而是“把治理定义收敛到可执行检查项”的问题。

这一轮的目标不是扩展能力，而是把门禁、留痕、演练、升级准入压缩成最小闭环，避免后续讨论继续发散成大而全制度文档。

## D. 最小门禁清单是什么

### D.1 最小门禁点

只保留 4 个最关键门禁点：

1. **任务入口门禁**
   - 判断任务是否应进入主线
   - 关注幂等、动作范围、基础可执行性

2. **执行前门禁**
   - 判断是否允许进入实际执行链
   - 关注 confirm 放行、前置条件、失败分层

3. **页面 / 执行门禁**
   - 判断页面是否可操作、可定位、可继续
   - 关注 controlled_page 证据、元素可达性、超时/缺失

4. **结果门禁**
   - 判断执行结果是否可确认、可回写、可留痕
   - 关注 success / verify_fail / 失败层归因

### D.2 最小验收标准

- 每个门禁点都能说清“在什么时机判断、判断什么、失败后落在哪一层”
- 每个门禁点都能对应到现有单动作主线中的具体节点
- 不要求新增实现，只要求定义口径清楚

---

## E. 最小留痕字段清单是什么

### E.1 `/tasks` 级别最小留痕

只保留任务级最小治理字段：

- 任务唯一标识
- 动作名
- 任务状态
- 最终结果
- 主失败层
- 主原因 / gate reason
- 是否回放过
- 是否留痕完成

### E.2 `/steps` 级别最小留痕

只保留步骤级最小证据字段：

- 步骤顺序
- 步骤名称
- 执行状态
- 失败码
- 页面证据字段引用
- 关键操作证据

### E.3 证据 / 演练记录最小留痕

只保留最小关联信息：

- 样本类型
- 样本日期
- 样本结论
- 回放入口
- 证据引用路径

### E.4 最小验收标准

- 能区分任务级结果与步骤级证据
- 能把失败层和主原因挂到同一任务上
- 能把回放和证据关联起来
- 不要求把所有字段一次性实现完，只要求口径最小且一致

---

## F. 最小演练 / 回归矩阵清单是什么

### F.1 固定样本

只保留 4 类固定样本：

1. `success`
2. `element_missing`
3. `page_timeout`
4. `verify_fail`

### F.2 每个样本最少要说明

- 触发方式
- 主断言
- 期望失败层
- 是否允许回放
- 是否进入 `/steps` 留痕

### F.3 最小验收标准

- 4 类样本都能映射到现有主线语义
- 每类样本都能说清主断言和失败层
- 不追求增加新样本，不追求覆盖所有边角

---

## G. 升级准入条件如何收敛

### G.1 最小准入条件

只保留 5 条最关键准入条件：

1. 门禁点已明确
2. 留痕口径已明确
3. 固定样本矩阵已明确
4. `--task-id` 回放路径已明确
5. 失败层 / 主原因映射已明确

### G.2 最小验收标准

- 能回答“什么条件下才值得讨论下一阶段”
- 能作为是否进入更接近生产或第二动作讨论的前置门槛
- 不要求形成正式制度，只要求形成可执行判断线

## I. 最小执行顺序是什么

如果下一轮继续推进，推荐顺序如下：

1. **先收敛门禁点清单**
   - 因为门禁决定治理切入点和失败归属

2. **再收敛留痕字段清单**
   - 因为留痕需要跟门禁和步骤证据对齐

3. **然后收敛演练 / 回归矩阵**
   - 因为样本要建立在门禁与留痕口径稳定之后

4. **最后收敛升级准入条件**
   - 因为准入条件依赖前面三类是否稳定

这个顺序的目标是先定边界，再定证据，再定样本，最后定升级门槛。
