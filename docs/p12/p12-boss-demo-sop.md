# P12-B 老板演示 SOP

## 一、演示目标

验证 P12-B 是否真正完成：

老板在飞书里搜索商品后，不需要再输入“加入监控第 N 个”，可以直接点击候选卡片里的“加入监控”按钮完成纳管。

同时保留文本方式，避免按钮回调不稳定时影响主链。

## 二、演示前提

需要启动：

1. B 项目：Ecom-Watch-Agent-Agent
2. A 项目 API / worker / 飞书 listener
3. 飞书机器人已在目标会话可用
4. P12-A 候选卡片展示已通过

B 默认地址：

http://127.0.0.1:8005

## 三、建议启动检查

### 1. 检查 B 服务

```bash
curl http://127.0.0.1:8005/health
如无 health，可用项目当前已有的 discovery 接口或 docs 里的启动方式确认。

2. 启动 A 主系统

按当前仓库实际命令启动。

常见方式：

uvicorn app.main:app --host 0.0.0.0 --port 8000
3. 启动 worker

按当前仓库实际命令启动。

常见方式：

celery -A app.workers.celery_app worker --loglevel=info
4. 启动飞书 listener

按当前仓库现有脚本或命令启动。

四、演示步骤
步骤 1：搜索商品

在飞书发送：

搜索商品：重力毯

预期：

先收到“已接收任务”
再收到候选结果卡片
卡片里展示前 3~5 条候选
每条候选旁边有“加入监控”按钮

通过标准：

卡片可读
query 正确
batch_id 正确
候选名称 / URL / 来源可见
每条候选都有按钮
步骤 2：点击按钮纳管

点击第 1 条候选的：

加入监控

预期：

系统补发文本，类似：

已加入监控。
- 选择编号：第 1 个
- 名称：xxx
- URL：https://xxx
- 对象ID：xxx
- 状态：active

通过标准：

不需要手动输入“加入监控第 1 个”
点击后能成功纳管
返回文本老板能看懂
不吐堆栈
不出现英文技术错误
步骤 3：验证文本方式仍可用

继续发送：

加入监控第 2 个

预期：

仍然可以通过旧文本方式纳管。

通过标准：

文本方式没有被按钮回调改坏
能成功加入监控
返回对象ID和状态
步骤 4：查看当前监控对象

发送：

看看当前监控对象

预期：

能看到刚才通过按钮或文本方式加入的对象。

通过标准：

对象存在
状态 active
名称 / URL 可读
步骤 5：失败样本

可选触发：

点击旧卡片按钮
点击非法 index
B 服务关闭后点击按钮

预期：

返回老板可读错误，例如：

未能加入监控。
原因：候选结果已失效，请重新搜索商品后再试。

通过标准：

不吐堆栈
不静默失败
提示用户如何继续操作
