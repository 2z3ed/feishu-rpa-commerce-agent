# P11-C 老板演示 SOP

## 一、演示目标

让老板在飞书里直接体验：

先搜索候选
→ 再按编号选择一个候选
→ 系统把它正式加入监控
→ 再查看监控对象确认新增已生效

## 二、演示固定样本

第一步飞书发送：
- 搜索商品：蓝牙耳机

第二步飞书发送：
- 加入监控第 2 个

第三步飞书发送：
- 看看当前监控对象

失败样本飞书发送：
- 加入监控第 2 个（无最近 discovery 上下文）

## 三、演示前准备

1. 启动 A 主系统
2. 启动 A 的飞书 listener
3. 启动 B 服务
4. 确认 B 运行在 127.0.0.1:8005
5. 确认 B 的以下接口可调用：
   - POST /internal/discovery/search
   - GET /internal/discovery/batches/{batch_id}
   - POST /internal/monitor/add-from-candidates
6. 确认 A → B HTTP 调用可用
7. 确认 Envelope 解包正常

## 四、演示步骤

### 第一步：在飞书里发搜索命令
例如：
搜索商品：蓝牙耳机

预期飞书返回：
- 搜索关键词
- 批次号
- 前 3~5 条候选列表
- 每条候选至少含：
  - 序号
  - 名称
  - URL
  - 来源

### 第二步：在飞书里发编号选择命令
例如：
加入监控第 2 个

预期飞书返回：
- 已加入监控
- 名称 / URL / 对象信息（若有）

### 第三步：验证新增对象
在飞书里发送：
看看当前监控对象

预期飞书返回：
- 当前监控对象列表
- 能看到新加入的对象（若 B 返回语义允许）

## 六、演示结论口径（冻结）

- 发现 → 编号选择 → 正式纳管 闭环已成立
- 无上下文失败路径已成立
- 当前未扩卡片交互
- 当前未做 pause / resume / delete
- 当前未切 PostgreSQL

## 七、演示成功标准

1. 飞书搜索命令能进入 A
2. A 能真实调到 B 的 discovery
3. 飞书能看到候选列表文本
4. 飞书编号命令能进入 A
5. A 能真实调到 B 的 add-from-candidates
6. 飞书能看到成功或失败文本
7. 如可行，“看看当前监控对象” 能看到新增对象
8. 不破坏 P10 / P11-A / P11-B 已收口链路
