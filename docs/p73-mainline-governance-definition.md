# P73 后续主线正式定义

> 状态：正式定义文档，非功能开发文档。  
> 适用范围：仅 `warehouse.adjust_inventory` 单动作主线。  
> 当前定位：单动作主线的门禁 / 留痕 / 演练深化。  
> 明确边界：仍不接正式生产，不接影刀控制台 / API Key / Flow，不让飞书直接触发影刀，不扩第二个动作，不做大重构。

---

## 1. 当前为什么进入这一阶段

P6.2、P70、P71、P72 都已经正式收口，说明当前项目已经不再处于“把单动作跑通”的阶段，而是进入“把单动作主线治理能力固定下来”的阶段。

当前已确认的事实是：

- 主链已经固定为：飞书消息 → 幂等 → 建任务 → Celery 异步执行 → LangGraph 编排 → API / confirm / RPA / bridge 执行链 → 回飞书结果 → `/tasks` 与 `/steps` 可查询、可追踪、可留痕
- 已经完成的收口阶段包括：
  - P6.2：Odoo `warehouse.adjust_inventory` 治理复制基线
  - P70：影刀本地 bridge PoC
  - P71：受控页面最小执行链
  - P72：更像真实后台的受控验证
- 当前仍然不是正式生产接入
- 当前仍然只围绕单动作主线，不扩第二个动作

因此，P72 之后最合理的下一步，不是继续做页面逼真化，也不是直接接生产，而是把现有单动作主线升级为一条**可稳定交接、可稳定回归、可稳定留痕、可稳定升级判断**的治理主线。

---

## 2. 为什么唯一推荐这条主线

P72 之后的唯一推荐主线是：**单动作主线的门禁 / 留痕 / 演练深化**。

原因是它同时满足四个条件：

1. **承接当前事实**：它直接建立在既有单动作主线上，不需要重建语义
2. **最小风险**：它不要求扩第二个动作，不要求接正式生产，不要求接影刀控制台侧能力
3. **最少返工**：它优先固化门禁、留痕、回放、演练矩阵，能减少未来升级时的返工
4. **最利于后续判断**：它能为未来是否进入更接近生产或是否复制第二动作，提供清晰准入条件

这条主线的价值，不是新增执行能力，而是把现有执行能力治理成可持续推进的基础设施。

---

## 3. 为什么不选其它候选项

### 3.1 不选“更严格的门禁 / 留痕 / 演练深化”之外的页面进一步受控验证

严格来说，这一项已经是当前唯一推荐主线的一部分。

如果把它再拆成“继续做更像真实后台的页面验证”，会重新把重点拉回页面逼真化，容易重复 P72 已经完成的工作，收益会越来越边际化。

### 3.2 不选“第二个动作复制验证”

不选的原因很直接：

- 会触碰当前明确冻结的边界：不能扩第二个动作
- 会引入新的动作语义、门禁语义、回放语义和留痕语义
- 很容易迫使回头修改当前单动作抽象，造成返工

### 3.3 不选“正式生产接入前的权限 / 运维 / 安全准备”

这条线后续一定会重要，但当前不是最优先主线。

原因是当前仍处于非生产接入语境，如果过早转向生产准备，会把治理升级和生产化准备混在一起，既增加范围，也增加返工风险。

---

## 4. 当前冻结边界

本阶段必须保持以下边界不变：

- 仍然只围绕 `warehouse.adjust_inventory`
- 仍然不接正式生产
- 仍然不扩第二个动作
- 仍然不接影刀控制台 / API Key / Flow
- 仍然不让飞书直接触发影刀
- 仍然不做大重构
- 仍然不回头修改 P6.2 / P70 / P71 / P72 已收口逻辑
- 仍然不新增生产接入逻辑

本阶段只允许做：主线定义、治理边界定义、留痕口径定义、演练与升级准入定义。

---

## 5. 这一阶段的核心目标

这一阶段的核心目标只有一个：

**把单动作主线固化成“可门禁、可留痕、可回放、可演练、可升级判断”的治理主线。**

具体含义是：

- 出现任务时，知道是否应该放行
- 执行时，知道在哪一层失败
- 结果出来时，知道如何回写和留痕
- 回放时，知道如何对照 `/steps` 和证据
- 后续升级时，知道是否已经满足准入条件

---

## 6. 这一阶段的最小交付物

本阶段的最小交付物不是功能集合，而是**治理定义集合**。

### 6.1 门禁点定义

至少要定义清楚以下门禁类型：

- 任务入口门禁
- 编排门禁
- 执行前门禁
- 页面门禁
- 结果门禁
- 回放门禁
- 治理门禁

### 6.2 留痕口径定义

至少要明确哪些字段属于主线必须留痕，哪些属于辅助留痕。

建议的归类方式是：

- `/tasks`：任务级结果和治理字段
- `/steps`：步骤级证据与失败码
- 证据目录：原始结果与回放关联
- 演练记录：样本、日期、结论、门禁命中情况

### 6.3 演练 / 回归矩阵主视图

至少要覆盖以下固定样本：

- `success`
- `element_missing`
- `page_timeout`
- `verify_fail`

并明确：

- 每个样本的触发方式
- 主断言
- 期望失败层
- 是否允许回放
- 是否进入 `/steps` 留痕

### 6.4 升级准入条件

至少要定义清楚什么条件下，才允许讨论更接近生产或第二动作复制验证。

---

## 7. 这一阶段明确不做什么

本阶段明确不做：

- 不扩第二个动作并直接实现
- 不接真实 Odoo 生产页面
- 不接影刀控制台 / API Key / Flow
- 不让飞书直接触发影刀
- 不做正式生产接入
- 不做大重构
- 不继续旧的错误 P73 文档路线
- 不拆出一堆 P73.1 / P73.2 / P73.3 开发轮次去发散实现

这轮只做正式主线定义，不做功能开发。

---

## 8. 进入后续更接近生产 / 扩第二动作前的升级准入条件

进入后续更接近生产或扩第二动作讨论之前，必须至少满足以下条件：

1. 单动作主线门禁点已稳定
2. `/tasks`、`/steps`、证据目录、演练记录口径已稳定
3. 固定样本回归矩阵已稳定
4. `--task-id` 回放与样本对照已稳定
5. 失败层 / 主原因映射已稳定
6. 不破坏当前冻结边界
7. 不需要推翻已有主线语义

只要这些条件没有满足，就不进入下一类更大范围的讨论。

---

## 9. 当前阶段结论

当前阶段的结论是：

**P72 之后，不应继续扩新动作，也不应直接接生产，而应先把单动作主线的门禁、留痕、演练和升级准入规则正式固化。**

这条主线是当前唯一推荐主线，也是后续是否能进入更接近生产、是否能讨论第二动作复制的前置基础。

---

## 10. 事实与建议的区分

为了避免把建议口径写成既成事实，本文件约定：

- **当前已确认事实**：P6.2 / P70 / P71 / P72 已收口，当前仍非生产接入，当前仍只围绕单动作主线
- **本阶段建议固化项**：门禁点、留痕口径、演练矩阵、升级准入条件
- **本阶段不作为事实陈述的内容**：尚未落地的字段、尚未实现的门禁、尚未完成的准入规则

---

## 11. 最终定位

P73 在这里被正式定义为：

**单动作主线的门禁 / 留痕 / 演练深化阶段。**

它不是生产接入轮，不是第二动作轮，也不是页面逼真化轮。

它的职责，是把 P72 之后的唯一合理方向，整理成一条边界清楚、可执行、可交接、可升级判断的正式主线。
