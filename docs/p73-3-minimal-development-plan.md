## 2. 最小开发清单

### 任务 1：固化门禁检查入口

**目标**：把门禁点从定义层收敛为可执行的检查入口，保证后续检查有统一入口，不散落在不同位置。

**为什么先做它**：门禁是前置条件。没有门禁入口，后续留痕、回归、升级判断都缺少统一起点。

**最小通过标准**：

- 门禁点可以被明确找到
- 门禁点名称、位置、职责保持一致
- 不影响现有主链行为

### 任务 2：整理留痕字段最小落点

**目标**：把 `/tasks`、`/steps`、证据目录、演练记录的字段归属整理成可执行的最小落点说明。

**为什么现在做**：留痕是回放和追踪的基础，且必须在门禁之后尽快稳定，以避免后续执行结果没有统一落点。

**最小通过标准**：

- 四类留痕载体的职责清晰
- 任务级、步骤级、证据级、演练级边界清楚
- 未实现字段仅作为建议项出现

### 任务 3：对齐固定样本回归矩阵

**目标**：把 `success`、`element_missing`、`page_timeout`、`verify_fail` 四个样本对齐到统一的检查结构。

**为什么现在做**：样本回归是验证门禁和留痕是否一致的最小闭环，必须在留痕之后尽快固定。

**最小通过标准**：

- 四个固定样本都能被明确定位
- 每个样本的触发方式、断言、失败层、回放权限一致可查
- 样本与 `/steps` 留痕口径对齐

### 任务 4：固化升级准入检查项

**目标**：把升级准入条件收敛为可执行的检查项，用于判断是否可以讨论更高一级主线。

**为什么最后做**：升级准入依赖门禁、留痕和回归矩阵的稳定性，必须放在这些基础项之后。

**最小通过标准**：

- 7 条准入条件都能被逐项检查
- 每条都有明确稳定性判断
- 不需要推翻已有主线语义

### 任务 5：建立最小执行顺序说明

**目标**：把前四项的执行顺序固化下来，减少后续开发返工。

**为什么要单列**：顺序本身就是降低返工的关键，如果顺序不明确，后续最容易先做错基础。

**最小通过标准**：

- 顺序清楚
- 每一步都有前置依赖关系
- 顺序符合单动作、非生产、非扩展边界

---

## 3. 最小实施顺序

建议顺序如下：

1. **先做门禁检查入口**
2. **再做留痕字段最小落点**
3. **再做固定样本回归矩阵对齐**
4. **最后做升级准入检查项**

### 为什么这个顺序最小且合理

- 门禁是入口，必须先稳定
- 留痕是结果落点，必须紧跟入口之后
- 回归矩阵依赖留痕和门禁，放在中间最合适
- 升级准入依赖前三者，应该最后收口

## 4. 每项任务的最小改动面

### 任务 1：固化门禁检查入口

**优先改动面**：

- 优先查看治理定义文档和验收模板文档是否已有统一入口描述
- 若进入实现，优先改动与任务入口、编排、执行前检查相关的定义位置

**先不要碰**：

- 已经正式收口的 P6.2 / P70 / P71 / P72 语义
- 第二动作相关任何文件
- 生产接入相关逻辑

### 任务 2：整理留痕字段最小落点

**优先改动面**：

- 优先查看 `/tasks`、`/steps`、证据目录、演练记录相关定义文档或字段说明
- 若进入实现，优先改动结果回写与步骤留痕的最小字段描述

**先不要碰**：

- 还未确认的扩展字段
- 生产权限、控制台、Flow 相关文件
- 大范围重构任务

### 任务 3：对齐固定样本回归矩阵

**优先改动面**：

- 优先查看样本定义、失败层映射、`--task-id` 回放相关说明
- 若进入实现，优先对齐四个固定样本的说明与检查结构

**先不要碰**：

- 新动作样本
- 真实生产样本
- 需要新增编排分支的大改动

### 任务 4：固化升级准入检查项

**优先改动面**：

- 优先查看升级准入条件清单、治理定义和验收模板
- 若进入实现，优先整理判断条件，不先动执行逻辑

**先不要碰**：

- 会推翻既有主线语义的改动
- 生产接入逻辑
- 第二动作路径

### 任务 5：建立最小执行顺序说明

**优先改动面**：

- 优先放在定义层文档中统一收口
- 若进入实现，仅作为开发计划说明，不直接影响运行逻辑

**先不要碰**：

- 任何执行链逻辑
- 任何实际门禁代码
- 任何新增生产相关路径
